# 29 王朝兴衰转盘：三司制适配施工计划

> 目标：借用“溥天之下”王朝兴衰的五段转盘、阶段迁移和催化因素展示思路，作为《变身大明》大明当前局势的总览。只复用框架和 DLC 美术，不接入天朝制、天命、朝贡圈、原运动集团或原具体数值。

## 1. 本体/DLC 核验结论

DLC 原机制位于：

```text
common/situation/situations/tgp_dynastic_cycle.txt
common/situation/catalysts/10_tgp_dynastic_cycle_catalysts.txt
common/script_values/10_tgp_dynastic_cycle_values.txt
common/scripted_effects/10_dlc_tgp_dynastic_cycle_scripted_effects.txt
common/scripted_triggers/10_tgp_dynastic_cycle_triggers.txt
common/on_action/dynastic_cycle_on_actions.txt
gui/window_tgp_dynastic_cycle.gui
gui/event_window_widgets/event_window_widget_situation_info_dynastic_cycle.gui
```

原转盘五阶段为“开疆拓土、政通人和、新朝征服、局势紧张、群雄割据”，以 situation takeover points 和 catalysts 推动。

但原实现不可直接拿来给三司制使用，因为它大量硬编码：

- `title:h_china` 及其 de jure 区域；
- `celestial_government`、天命合法性与天朝官僚参数；
- 朝贡/册封、丝绸之路、帝国国库和科举计时；
- `advancement/expansion/conservative/pro_hegemon` 等 participant groups；
- GUI 中 `GetSituation('dynastic_cycle')` 和专用 `SituationDynasticCycleWindow` C++ 接口。

重定义 `dynastic_cycle` 会与 DLC 键冲突；把 `h_greatming` 冒充 `h_china` 会破坏兼容边界。因此不启动、不覆写、不修改 DLC 原 situation。

## 2. 采用的轻量结构

不用第二套 situation，只新增一个“查看大明局势”决议和带自定义 widget 的信息事件：

```text
common/decisions/minghm_outlook_decisions.txt
common/scripted_values/minghm_outlook_values.txt
common/scripted_effects/minghm_outlook_effects.txt
events/minghm_outlook_events.txt
gui/event_window_widgets/event_window_widget_minghm_outlook.gui
localization/simp_chinese/minghm_outlook_l_simp_chinese.yml
```

这样既能显示转盘，又不覆盖 HUD、`window_tgp_dynastic_cycle.gui` 或公共 situation 窗口。决议无费用、无冷却，仅对当前 `h_greatming` 持有者显示。

## 3. 五段转盘映射

直接显示 `25` 已有 `minghm_historical_stage`，不新增另一份长期“王朝阶段”：

| stage | 三司制名称 | 政治含义 | 转向 |
|---:|---|---|---|
| 0 | 秩序稳固 | 三司—中央体系能够维持日常统治，守成力量占优 | 走向改革竞争或秩序失衡 |
| 2 | 改革竞争 | 改革与守成力量都已组织化，政策博弈最活跃 | 改革成功后整合，失败后失衡 |
| 1 | 秩序失衡 | 行政、战争或党争压力侵蚀执行能力 | 可修复、转入改革，或滑向双重权力 |
| 3 | 双重权力 | 朝廷外动员/军政力量足以争夺实际支配权 | 镇压、妥协、革命或新秩序整合 |
| 4 | 新秩序整合 | 一条改革/革命/反革命路线胜出，正在制度化 | 可稳定为新常态，也可重新失衡 |

GUI 按 `0 → 2 → 1 → 3 → 4 → 0/2` 排列成环。阶段仍可逆，不把王朝兴衰写成固定年月倒计时。

## 4. 三条局势分数

打开转盘时即时计算，范围钳制 `0..200`，只用于显示和阶段候选判定：

```text
秩序整合力 = state_capacity
           + elite_cohesion
           + (100 - crisis_pressure) / 2

变革推动力 = reform_momentum
           + popular_mobilization / 2
           + commercial_influence / 4

秩序解体力 = crisis_pressure
           + military_autonomy / 2
           + conservative_power / 2
           - state_capacity / 2
```

原模组皇权只作为转盘提示的临时修正：皇权强时显示“皇帝仍有强推余地”，皇权弱时显示“诏令难以越过官僚与集团阻力”；不把 `huang_quan_value` 并入每年自动分数，避免重复驱动。

每条分数展示前三项来源。它们不是新的存档经济数据，事件关闭即清除显示缓存。

## 5. 阶段候选与防抖

年度只判定一次：

| 候选阶段 | 建议门槛 |
|---|---|
| 秩序稳固 | 整合力 `≥130`、解体力 `<80`、改革推动力 `<110` |
| 改革竞争 | 改革推动力 `≥100`、解体力 `<120` |
| 秩序失衡 | 解体力 `≥90`，但尚未达到双重权力条件 |
| 双重权力 | 解体力 `≥140`、精英协调 `≤40`，且民众动员或军事自主 `≥60` |
| 新秩序整合 | 当前为双重权力/改革竞争，整合力或改革推动力 `≥130`，解体力 `<120`，并完成重大危机结算 |

候选优先级：双重权力 > 新秩序整合 > 秩序失衡 > 改革竞争 > 秩序稳固。连续两次年度判定相同才切换，沿用 `minghm_stage_candidate` 防抖。

阈值是本项目初始平衡值，不复制 DLC 的 `2000/4000` 天命 takeover points。玩家测试后只调整本表，不改 DLC 数值。

## 6. 催化因素的融合方式

保留 DLC “事件推动转盘”的思想，不定义 situation catalyst 对象。已有事件结算调用统一全国变量 effect：

| 催化类别 | 示例 | 典型影响 |
|---|---|---|
| 积极行政 | 改革充分落实、危机妥善解决 | 国家能力/改革势头 `+3..8` |
| 政治阻塞 | 政策纸面落实、党争瘫痪 | 守成力量/危机压力 `+3..8` |
| 战争冲击 | 战败、首都受威胁、军队坐大 | 危机压力/军事自主 `+5..10` |
| 继承冲击 | 新君、摄政和内外廷争权 | 精英协调 `-3..10` |
| 组织动员 | 民众、军队或商业力量进入政治 | 对应长期值 `+3..8`，同时产生对价 |
| 新秩序承诺 | 改革/革命胜利后的制度化 | 整合力上升，但必须履行承诺 |

转盘只是读取和解释这些结果，不额外重复修改数值。

### 自定义布局

- 轮盘底图和边框居中；
- 五个阶段图标按固定位置排成环；
- 当前阶段图标显示本体金色进度光圈，其他阶段降透明度；
- 中央显示当前阶段名、简述和“最可能下一阶段”；
- 下方三条进度条显示整合力、改革推动力、解体力及具体数字；
- 右侧列出最近三次造成长期值变化的原因（若历史缓存存在）；
- 不显示 DLC 运动集团、天命、地图模式、朝贡、合法性或行动按钮。

DLC 原轮盘旋转依赖专用 C++ `SituationDynasticCycleWindow`，不能安全套到自定义数据。因此 MVP 用固定轮盘和阶段高亮，不复制旋转动画；视觉仍是转盘，性能与兼容更稳。

## 8. 事件 widget 数据传递

决议触发 `minghm_outlook.0001`。事件 immediate：

1. 从 `title:h_greatming` 读取 stage 与八项值；
2. 在玩家人物上设置带 `minghm_outlook_cache_*` 前缀的临时显示变量；
3. widget 通过 `GetPlayer.MakeScope.Var(...)` 读取，这是本体 `child_examination_success_chance.gui` 已使用的可行模式；
4. 唯一“关闭”选项调用 `minghm_clear_outlook_cache_effect` 删除全部缓存。

每次打开前先执行清理再重算，旧存档或强制退出留下缓存也不会累积。人物死亡与年度维护再做一次兜底清理。缓存不参与实际阶段判定。

## 9. 性能与兼容

- 不创建 `situation`、sub-region、participant 或 movement；
- 不执行月度 pulse，不扫描地图/人物；
- 年度阶段判定只在 `h_greatming` 上一次；
- GUI 仅玩家主动打开时计算三条分数；
- 三司制政府无需伪装 `celestial_government`；只校验 `h_greatming` holder；
- DLC 缺失时本项目本就不满足硬依赖，不制作无 DLC 美术回退包；
- 不覆写 `window_tgp_dynastic_cycle.gui`、`hud.gui` 或 `dynastic_cycle` situation。

## 10. 验收

- 三司制大明能通过决议打开转盘，非明不可见；
- 无 `title:h_china`、mandate、tributary、movement 或 celestial government 读取；
- 五阶段高亮、三条分数、下一阶段提示与年度实际阶段一致；
- 关闭后所有 `minghm_outlook_cache_*` 清除；
- 连续打开 100 次不增加人物、长期变量或 situation；
- 同时存在 DLC 原 `dynastic_cycle` 时不发生键/GUI 冲突；
- 100 年观察只有年度一次阶段判定，无月度新增成本。
