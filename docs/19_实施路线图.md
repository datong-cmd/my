# 实施路线图

> 目标：把完整策划拆成可以由人或 AI 逐批生成、逐批实机验证、逐批回滚的 CK3 Mod 交付路线。
>
> 原则：先证明状态载体、调度、AI、存档和原生接口，再扩内容；每一阶段必须形成可玩的闭环，不按文件类型横向铺满空壳。

## 1. 总体路线

```text
M0 技术探针与脚手架
→ M1 清丈—区域—阶级—危机垂直切片
→ M2 权力运行核心
→ M3 财政、土地、市场与人口
→ M4 军事、军工与战争政治
→ M5 藩属、边疆与跨政权
→ M6 再工业化、资本与制度竞争
→ M7 革命、反革命与重大事件导演
→ M8 内容扩充、平衡、兼容与发布
```

这不是线性剧情。路线图的线性只表示代码依赖，游戏中的历史阶段仍可混合、倒退、复辟和分裂。

## 2. 每阶段共同完成条件

一个里程碑只有同时满足下列条件才能进入下一阶段：

- 有玩家入口、AI 入口、成功、失败、主动回撤和清理；
- 公共接口已登记在 `18A`，事件已登记，未越权写变量；
- 新游戏、继承、存读档和旧 schema 迁移通过；
- 缺 DLC/不适用政权按设计降级；
- error.log 没有新增解析、scope 和未定义键错误；
- 自动检查和本阶段回归矩阵通过；
- 观察局没有对象泄漏、事件风暴和明显单一路线垄断；
- 文档、索引和版本说明同步；
- 已推送一个可回滚的 Git 提交/标签。

## 3. M0：技术探针与最小脚手架

### 3.1 目的

不用先写玩法，验证所有高风险 CK3 接口是否按预期工作。M0 失败时应修改适配方案，而不是让几十个机制建立在猜测上。

### 3.2 实现范围

- Mod descriptor、版本、依赖和首个 game rule；
- `minghm` namespace 与目录骨架；
- `core/adapter/debug` 三模块；
- 目标国家/主头衔识别；
- DLC feature 检测与无 DLC 提示；
- additive on_action 注册；
- state title 持久变量与 schema 1；
- situation 实例能否保存变量、参与组和阶段；
- 原生国家国库读写；
- 原生功绩/任命/家产至少各做一个只读或可逆探针；
- 保存、读取、继承后的 scope 持久性；
- debug decision：初始化、dump、验证、清理测试档。

### 3.3 不做

- 不写正式平衡数值；
- 不写十三类历史事件；
- 不建自定义大 GUI；
- 不复制参考模组代码；
- 不处理所有兼容 Mod。

### 3.4 退出条件

| 探针 | 通过定义 | 失败后的固定替代 |
|---|---|---|
| state title 载体 | 继承、失位、存读档仍正确 | 重新定义稳定主头衔映射 |
| situation 变量 | 创建、读写、存档、关闭均正确 | state title 固定危机槽 + situation 仅显示 |
| on_action | 仅触发一次、无同名覆盖 | 使用已确认的合并/嵌套入口 |
| 国库 adapter | 增减和 tooltip 与原生一致 | 暂停依赖功能，不另造重复国库 |
| DLC 缺失 | 安静禁用，不刷错 | game rule 强制阻止目标玩法 |
| schema | 重复初始化无副作用 | 修正幂等与迁移后再继续 |

## 4. M1：首个可玩垂直切片

详细边界见 [19A_MVP垂直切片.md](19A_MVP垂直切片.md)。核心闭环：

```text
国家初始化
→ 8 区域结构基线
→ C1—C8 阶级基线
→ 玩家/AI 提出土地清丈
→ 选择联盟与试点区域
→ 地方执行变形、士绅反制和财政收益
→ 财政—地方阻力危机 K0—K6
→ 改革成功/妥协/失败/回撤
→ 区域、阶级、联盟、阶段与记忆年度结算
```

### 4.1 必须证明

- 一个政策既能提高国家能力，也能把负担转嫁给弱势群体；
- C2、C3、C7 不是固定同盟；
- 执行能力和政治支持分开；
- AI 不读取隐藏的地方造假精确值；
- 改革失败仍产生新局面，而非简单 game over；
- 二十年观察局能出现至少三种可解释结果。

### 4.2 退出条件

- 玩家能在 30 分钟内理解并完成一次改革—危机循环；
- AI 在不同初始条件下会启动、妥协、强推或放弃；
- 20 年多种子测试无重复扣费、幽灵槽和年度多结算；
- 性能相对本体无可见事件风暴，初步计时进入预算。

## 5. M2：权力运行核心

### 5.1 内容

- 皇权生命周期、内阁席位、票拟、封还和首辅/摄政；
- 六联盟可见组织与领袖代理；
- 流官、科举候选池、授官、考成和任期；
- 六科、都察院、刑部、大理寺的案件流水线；
- 锦衣卫式非常监察，作为信息/强制的高收益高俘获工具；
- 职位、承诺、弹劾、调查与任命的共同政治预算；
- A1—A6 路线惯性、兑现、背叛和领导层替换。

### 5.2 垂直切片顺序

1. 一个关键任命：候选池—联盟支持—皇权决断—考成；
2. 一个政策封还：内阁拟议—言官反对—妥协/强推；
3. 一个重大案件：调查—部院审理—复核—政治结算；
4. 一个非常监察案件：情报收益—诏狱成本—机构自主化风险。

### 5.3 核心平衡

- 皇权直接控制提高短期速度，但提高公私混同和继承断裂；
- 官僚规则提高持续能力，但会产生刚性、造假与自我保护；
- 士绅网络降低地方治理成本，也会隐藏税基和阻滞统一；
- 非常监察改善信息与威慑，也会制造诬告、恐惧和独立机构利益。

### 5.4 暂缓

不在 M2 制作完整官职百科、所有历史人物和每个部院的独立小游戏。

## 6. M3：财政、土地、市场与聚合人口

### 6.1 内容

- 公共国库、内帑、预算承诺和信用；
- 五类关键库存仅在确有玩法时实现，不做通用仓库模拟器；
- 赋役、清丈、土地、矿业、基础设施、地方留成；
- 私产、继承、地契、商帮、会馆、信用和特许；
- 商品化、债务失地、工资劳动、城市化和迁徙的聚合转化；
- 区域冲击传播、年度结算和五年校准；
- 土地/市场/劳动至少三条改革路线。

### 6.2 推荐切片

1. 战争预算：公共国库与内帑谁支付；
2. 粮运中断：运输、粮食安全、商人信用和地方截留；
3. 海贸开放：沿海增长、内地分化、特许垄断和走私；
4. 债务失地：C5/C2 获益、C7→C8 转化和抗租危机；
5. 基础设施：建设、维护、俘获和跨区域溢出。

### 6.3 退出条件

- 高增长可伴随高不平等和国家能力下降；
- 增税不等于实际入库，政策执行变形可解释；
- 不同所有制的同一工业项目产生不同阶级后果；
- 人口只用聚合值，无人物生成膨胀。

## 7. M4：军事、军工与战争政治

### 7.1 内容

- 军功生成、核验、排队和兑现；
- 卫所/军户衰变、募兵、家丁、将门和受薪军队竞争；
- 五军都督府式军事权力入口与文武冲突；
- 军饷、军粮、装备、训练、运输的有限状态；
- 新兵种/火器能力通过原生单位与修正表达；
- 军工所有制：国家、宫廷、军镇、商资、作坊联合；
- 战争紧急权力、军头化、兵变和复员；
- 原生战争/军队结果进入 C4、财政与阶段。

### 7.2 推荐切片

1. 欠饷—兵变危机；
2. 军功兑现—官职/土地/金钱三选；
3. 国家军器局—供给闭环—质量/腐败；
4. 战时征发—短期产能—长期生产破坏；
5. 胜利将领—军政威望—去军头化或政变。

### 7.3 退出条件

- 单纯堆军费不会自动得到可靠军队；
- 私人军队和国家军队各有短期优势和长期成本；
- 战争结果改变阶级力量、财政与动员方式；
- 和平期仍有复员、军户土地和军工订单问题。

## 8. M5：藩属、边疆与跨政权

### 8.1 内容

- 原生贡属与独立属国桥接；
- 礼仪、贡赐、军事义务、贸易和干预的多维关系；
- 土司继承请封、盟约、归流、地方知识和叛乱风险；
- 殖民/移民边疆的土地、劳动、财富输送和族群政治抽象；
- 征服型军事共同体、八旗式供养/从征/改革的通用结构；
- 中心危机对外围自治和竞争中心的传播。

### 8.2 切片

1. 一个土司继承危机；
2. 一次贡赐赤字与边防义务谈判；
3. 一项可逆归流试点；
4. 一次移民开发引发的土地—劳动力冲突；
5. 一个征服共同体的供养危机与内部改革。

### 8.3 边界

不为每个属国创建独有资源系统；使用关系类型、危机和区域结构组合表达差异。

## 9. M6：再工业化、资本与制度竞争

### 9.1 内容

- 工业七环：资本、原料/能源、技术、劳动、运输、需求、再投资；
- 项目槽与多种所有制；
- 商业资本投地产/贸易/生产的选择；
- 包买制、分散工场、集中工场和国家工业；
- 工资劳动、劳动保护、失业、组织化和资本外逃；
- 技术教育、标准化、采购、军工和民用扩散；
- 官僚发展、商人寡头、宫廷工业、军政工业、合作/公共路线；
- P2/P3/P6/P7 的阶段组合，而非单科技解锁。

### 9.2 推荐切片

1. 矿冶—军器闭环；
2. 纺织包买制—城市工场转化；
3. 商业信用危机；
4. 劳动保护—工资/就业/资本迁移；
5. 国家订单撤回后的去工业化；
6. 多所有制争夺同一基础设施。

### 9.3 退出条件

- 工业增长没有固定“现代化胜利”buff；
- 资本能崛起，也能投地产、寡头化、破产或与官僚/军队结合；
- 劳动集团由物质条件和组织经验形成，不由年份生成；
- 生产力提高与生产关系冲突可分离并进入政治。

## 10. M7：革命、反革命与重大事件导演

### 10.1 内容

- 动员阶梯、组织网络和双重权力；
- 改革失败、国家危机、精英分裂、群众组织与替代能力组合；
- 反革命资源、社会联盟和强制整合；
- 内战/革命胜负后的国家、军队、财政、粮食和合法性整合；
- 复辟、妥协宪制、军政国家、大众参与、碎片化等多结局；
- 十三类明代事件族与十年结构情景包；
- 事件导演按结构压力、冷却、认知预算和历史人物替代选取。

### 10.2 实施顺序

1. 非革命民变：请愿—抗租—地方妥协/镇压；
2. 精英政变/强制整合；
3. 双重权力 situation；
4. 革命与反革命的资源动员；
5. 战后整合与复辟；
6. 把十三类事件族逐批接入，而非一次写完。

### 10.3 退出条件

- 高不满但低组织不会自动革命；
- 革命胜利不等于治理成功；
- 镇压能短期奏效但保留组织、仇恨与生产后果；
- 各结局均能继续至少二十年玩法；
- AI 能根据物质利益、组织承诺和有限信息站队。

## 11. M8：内容、平衡、兼容与发布

### 11.1 内容

- 扩写事件变体和人物文本，但不新增未经验证的底层系统；
- 历史开局参数与架空沙盒参数；
- 教程、百科、趋势说明和信息层级；
- 复用本体/“溥天之下”图标、插画、模型和音乐的清单化接入；
- 可选参考模组兼容桥，保持零通用路径覆盖；
- 100/200 年观察局、多种子平衡和玩家测试；
- 简体中文完整，其他语言以占位或社区翻译边界处理；
- 版本、迁移、已知问题和卸载/停用说明。

### 11.2 发布梯级

| 渠道 | 内容边界 | 进入条件 |
|---|---|---|
| 开发构建 | 调试工具、可能破档 | 静态解析通过 |
| 内部 alpha | M1—M3 主要闭环 | 存档/AI/二十年观察通过 |
| 封闭 beta | M1—M7，内容未完全 | 百年观察、主要路线可玩 |
| 公开 beta | 完整功能，允许平衡调整 | 迁移和发布阻断项清零 |
| 1.0 | 稳定契约和内容密度 | 兼容、文档、长跑与玩家反馈收口 |

## 12. 依赖矩阵

| 系统 | 先决模块 | 可并行内容 | 不得提前依赖 |
|---|---|---|---|
| 阶级 | core、region 基线 | 联盟本地化 | 革命事件 |
| 联盟 | class、人物代理 | 官僚/军事组织文本 | 完整资本系统 |
| 阶段 | class、region、crisis 缓存 | 事件叙事 | 线性科技门槛 |
| 改革 | politics、region、fiscal adapter | 多种改革档案 | 革命结局 |
| 危机 | situation/槽探针、class、politics | 类型叙事 | 十三类一次性全接入 |
| 军工 | fiscal、project、原生军事 adapter | 美术/单位定义 | 世界人口模拟 |
| 资本 | market、project、class | 行商/会馆事件 | 现代公司法全模拟 |
| 革命 | stage、class、alliances、crisis | 文本变体 | 固定年份触发 |

## 13. 高风险项目与决策点

| 风险 | 最晚决策阶段 | 验证办法 | 失败方案 |
|---|---|---|---|
| situation 动态实例/变量不适合多危机 | M0 | 创建—保存—阶段—关闭探针 | 固定槽 + situation 显示 |
| 原生国库 API 版本变化 | M0 | adapter 单元探针 | 暂停依赖，不造双国库 |
| 八区域 anchor 在不同开局不稳定 | M1 | 多开局映射测试 | 功能区按 title flags 重建 |
| AI option value 过于昂贵 | M1 | profiler/调用计数 | 季度缓存、候选集缩小 |
| 原生 GUI 扩展冲突 | M1/M2 | 覆盖扫描、多 Mod 启动 | 独立轻量窗口/事件降级 |
| 官职与任命系统复杂耦合 | M2 | 单一任命垂直切片 | 只读原生职务 + 自有政治层 |
| 人口/市场过度模拟 | M3 | 100 年对象/存档增长 | 更强聚合，删除库存细分 |
| 自定义兵种破坏战争平衡 | M4 | 原生军队对照 | 先用能力/修正，不新兵种 |
| 革命导致不可继续 | M7 | 每结局 20 年续玩 | 强制建立最低国家/地方载体 |

## 14. 内容生产规则

- 先写机制档案，再写事件树；每个事件必须映射一个结构状态或承诺。
- 同一结构事件至少有三套人物/地区文本变体，但共用结算接口。
- 历史人物提供加权与独特文本，不作为必要 scope。
- 史实日期只影响权重和氛围，不绕过结构条件硬触发。
- 每新增十个叙事事件，至少跑一次事件认知预算和重复率检查。
- 美术先复用已登记原生资源；新美术不阻断核心机制。

## 15. 版本与 Git 策略

- `main` 始终保持可加载、文档与代码契约一致；
- 一个垂直切片一个主题提交，避免“全系统初始化”巨型提交；
- 公共接口变更单独记录，公开后遵循弃用—迁移—删除流程；
- 每个可玩里程碑打版本标签，保存对应观察局种子与基线；
- 不提交本体、DLC、创意工坊源文件或未获授权资产；
- 生成索引和测试报告可提交，临时日志与大型存档放测试工件而非仓库历史。

## 16. 开发优先级判断

遇到“要不要现在做”时依次问：

1. 是否完成当前垂直切片的闭环？
2. 是否验证一个高风险原生接口？
3. 是否增加新的真实选择、反制或失败可玩性？
4. 是否减少后续 AI 生成代码的歧义？
5. 是否在性能和存档预算内？

若五项均否，推迟到内容扩充阶段。详细历史官名、稀有地区事件和美术变体不能挤占状态、AI、存档与清理工作。
