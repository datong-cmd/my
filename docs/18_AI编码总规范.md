# AI 编码总规范

> 文档性质：新 Mod 的冻结编码契约。后续 AI 生成脚本时，本文件优先于机制文档中的示例名称；设计含义仍以对应机制文档为准。
>
> 目标版本：Crusader Kings III `1.19.0.6 (Scribe)`；主要 DLC 接口为“溥天之下”。
>
> 关联文档：[数据字典](18A_变量状态与接口数据字典.md)、[事件与本地化模板](18B_事件触发效果与本地化模板.md)、[技术映射](13_CK3本体与溥天之下技术映射.md)、[性能预算](17A_性能缓存存档与测试预算.md)。

## 1. 这份规范解决什么

策划资料允许讨论多种路线，代码不能同时保留多套命名和状态来源。本规范冻结：

- 唯一前缀、事件编号、目录和数据库键；
- 模块所有权、跨模块调用方向和公共接口；
- 状态源、缓存、更新频率和存档迁移；
- AI 每次生成代码必须获得的上下文和禁止自行决定的事项；
- 日志、调试、错误恢复、兼容和验收标准。

它不是要求一次写完整个 Mod。正确方式是按垂直切片实现，每次只扩展已登记接口。

## 2. 规则优先级

发生冲突时，从高到低采用：

1. 用户在当前实现任务中的明确要求；
2. `18`、`18A`、`18B` 的冻结契约；
3. `17A` 的性能、存档与测试硬门槛；
4. `13`、`13A` 的本体接口、版本和兼容边界；
5. `14`—`17` 的新 Mod 机制设计；
6. `05`—`12B` 的参考模组分析；
7. `01`—`03` 的自动索引；
8. AI 的一般 CK3 经验或未经本地脚本确认的猜测。

自动索引只能证明“存在”，不能证明运行时可达。任何不确定的本体语法必须标记 `R-待实机验证`，不得伪装为已确认。

## 3. 全局命名冻结

### 3.1 唯一前缀

| 对象 | 冻结格式 | 示例 |
|---|---|---|
| 事件命名空间 | `minghm` | `namespace = minghm` |
| 事件 ID | `minghm.<整数>` | `minghm.1201` |
| 数据库键 | `minghm_*` | `minghm_reform_land_survey` |
| 变量 | `minghm_*` | `minghm_land_concentration` |
| 全局变量 | `global_var:minghm_*` | `global_var:minghm_schema_version` |
| 保存 scope | `minghm_*` | `save_scope_as = minghm_state_title` |
| 修正 | `minghm_*_modifier` | `minghm_tax_resistance_modifier` |
| 标志 | `minghm_*_flag` | `minghm_crisis_locked_flag` |
| 本地化键 | `minghm_*` | `minghm_crisis_fiscal_title` |
| GUI 类型 | `minghm_*` | `minghm_state_panel` |
| 图像资源 | `GFX_minghm_*` | `GFX_minghm_icon_class_c5` |
| 音效/音乐引用 | `minghm_*` | `minghm_sfx_memorial_open` |
| 文件名 | `minghm_<模块>_<用途>.txt` | `minghm_class_scripted_effects.txt` |

禁止使用宽泛的 `ming_*`，也禁止沿用参考模组或本体的定义键。`minghm` 的含义只作为稳定技术前缀，不在玩家文本中显示。

### 3.2 标识符语法

- 全部 ASCII 小写 `snake_case`；不得使用拼音声调、空格、连字符或中文键名。
- 名词放前，动作放后：`minghm_class_c5_power`、`minghm_start_crisis_effect`。
- 布尔值使用 `has/is/can/needs`；数值不使用 `is_`。
- 跨模块公共接口必须以 `_trigger`、`_value`、`_effect` 结尾。
- 事件 option 使用 `a/b/c...`，语义由本地化和调用的 effect 表达，不在 ID 后拼长句。
- 不把阈值写进名称；阈值由命名脚本值或常量接口维护。
- 同一概念不得出现同义键，例如 `bureaucratic_capacity` 与 `admin_capacity` 二选一；冻结字典中采用前者就不再新增后者。

## 4. 事件号段

单一 namespace 下按模块分配号段。每个号段保留空洞，禁止为了连续编号而移动已发布事件。

| 号段 | 模块 | 典型用途 |
|---:|---|---|
| 1—99 | 初始化与迁移 | 新游戏、旧存档升级、恢复入口 |
| 100—199 | 调试与测试 | 仅调试模式可见的诊断事件 |
| 200—399 | 国家与历史阶段 | 主头衔初始化、阶段变化、王朝周期桥接 |
| 400—699 | 阶级与政治联盟 | 画像、联盟承诺、领导层、背叛记忆 |
| 700—999 | 区域与生产关系 | 区域初始化、年度结算、迁徙、冲击传播 |
| 1000—1199 | 财政、土地与市场 | 预算、清丈、信用、商业和赋役 |
| 1200—1399 | 官僚、科举与监察 | 任命、考成、弹劾、司法与信息 |
| 1400—1599 | 军事、军工与战争政治 | 军饷、军功、军工、军头化 |
| 1600—1799 | 藩属、边疆与跨政权 | 朝贡、土司、殖民、征服共同体 |
| 1800—2099 | 改革与工程 | 议程、试点、扩散、回撤、工业项目 |
| 2100—2499 | 危机框架 | 通用 C0—C6、谈判、承诺与结算 |
| 2500—3199 | 明代事件族 | 十三类情景包及其组合事件 |
| 3200—3499 | 革命、反革命与复辟 | 动员、双重权力、内战后整合 |
| 3500—3699 | AI 与记忆 | AI 评估、路线审查、组织记忆 |
| 3700—3899 | UI 辅助事件 | 面板跳转、教程、摘要，不做核心计算 |
| 3900—3999 | 兼容桥 | 版本或可选兼容，不直接覆盖他 Mod |
| 4000—4999 | 扩展保留 | 首版禁止使用，避免过早耗尽空间 |

事件登记表必须记录：ID、文件、触发入口、根 scope、是否只触发、冷却、幂等锁、后续事件、所属测试。

## 5. 目录与模块所有权

### 5.1 推荐目录

```text
common/
  character_interactions/
  decisions/
  game_rules/
  modifiers/
  on_action/
  scripted_effects/
  scripted_guis/
  scripted_modifiers/
  scripted_rules/
  scripted_triggers/
  scripted_values/
  situation_types/
  story_cycles/
events/
gui/
localization/simp_chinese/
gfx/interface/icons/minghm/
```

只有经 `13` 确认存在且本切片需要的目录才创建。不得复制参考模组完整目录树作为脚手架。

### 5.2 逻辑模块

| 模块键 | 唯一职责 | 拥有的持久状态 | 可调用 |
|---|---|---|---|
| `core` | 初始化、迁移、主头衔识别、调度 | schema、初始化、tick 标记、功能开关 | 本体适配、所有模块公共入口 |
| `adapter` | 包装本体/DLC 接口 | 仅兼容检测；不拥有玩法状态 | 本体与 DLC 原生 effect/trigger/value |
| `class` | C1—C8 七维状态与全国派生 | 阶级原始维度、缓存、记忆 | region、politics、crisis |
| `politics` | 六联盟、领袖、承诺、国家议程 | 联盟支持、凝聚、路线记忆 | class、reform、crisis |
| `region` | 八区域八轴、生产关系和传播 | 区域轴、dirty、上次结算 | adapter、class |
| `stage` | P0—P9 与四层制度 | 当前阶段、制度档位、滞后 | class、region、crisis |
| `reform` | 改革生命周期与工业工程 | 活跃改革/项目、里程碑、反制 | politics、region、fiscal |
| `crisis` | C0—C6 通用危机容器 | 危机槽、仪表、参与组、结算记忆 | 其他模块公共接口 |
| `fiscal` | 公共财政、内帑、信用和预算桥 | 仅新抽象值；原生国库由 adapter 读写 | adapter、crisis |
| `military` | 军饷、军工、战争政治 | 军工能力、忠诚/私有化风险 | adapter、fiscal、crisis |
| `frontier` | 藩属、边疆与跨政权 | 关系记忆、边疆模式 | adapter、crisis |
| `ai` | 有限信息评估、效用、路线惯性 | AI 路线/承诺/记忆 | 所有模块只读接口 |
| `ui` | 显示、提示、教程和快捷操作 | 不拥有核心状态 | 只读 values/triggers；操作调用公共 effects |
| `debug` | 检查、修复、观察局工具 | 调试日志标记 | 所有模块诊断接口 |

### 5.3 写入权

- 一个持久变量只有一个所有模块。
- 非所有者不得直接 `set_variable/change_variable/remove_variable`；必须调用所有者 effect。
- UI、事件 option 和 AI 不直接改底层变量，只调用公共 effect。
- `adapter` 是本体资源的唯一写入门面，例如天朝国库、原生功绩或局势参与组。
- 缓存变量只有重算 effect 写入；业务事件只置 dirty。
- 调试修复可越权，但必须放在 `debug` 文件并明确输出日志。

## 6. 依赖方向

允许方向：

```text
ui / ai / events
        ↓
public triggers + values + effects
        ↓
class / politics / region / stage / reform / crisis / fiscal / military / frontier
        ↓
core scheduler + adapter
        ↓
CK3 base game / All Under Heaven DLC
```

禁止：

- `adapter` 反向调用玩法模块；
- `class` 与 `region` 在同一调用链互相重算；
- UI 中复制业务公式；
- on_action 中嵌入数百行计算；
- 事件 A 直接写事件 B 所属模块变量；
- 以本地化文本、事件标题或 GUI 可见状态作为逻辑真值。

循环关系通过两阶段更新处理：先应用增量并置 dirty，再由季度/年度调度统一聚合。

## 7. 文件职责

每一模块至少按下列文件拆分，不强迫为每个小机制新建文件：

```text
common/scripted_triggers/minghm_<module>_triggers.txt
common/scripted_values/minghm_<module>_values.txt
common/scripted_effects/minghm_<module>_effects.txt
events/minghm_<module>_events.txt
localization/simp_chinese/minghm_<module>_l_simp_chinese.yml
```

- trigger 只回答条件，不产生副作用。
- value 只计算数字，不设置变量。
- effect 执行原子或事务性变更。
- event 负责叙事、scope 捕获、玩家选择和调用 effect。
- on_action 只做资格筛选、调度和调用入口。
- modifier 表达持续影响，不充当复杂数据库。
- decision/interaction 只做可用性、成本、AI 门面，不复制底层结算。

## 8. 状态、缓存与派生值

### 8.1 三类数据

| 类别 | 定义 | 示例 | 规则 |
|---|---|---|---|
| 源状态 | 玩家行动或结算直接改变 | 区域土地集中、联盟承诺 | 必须持久保存并指定写入者 |
| 缓存 | 由多个源状态聚合 | 全国 C5 力量、国家应对力 | 带 dirty；按预算重算 |
| 即时派生 | 读取时低成本算出 | 某政策的当前支持分 | 不持久化，除非性能测量证明需要 |

同一量不得既作为源状态又由公式覆盖。重构时必须通过 schema 迁移删除旧来源。

### 8.2 数值约定

- 标准仪表范围 `0..100`，所有公共变更 effect 都要 clamp。
- 权重和占比默认 `0..100`，脚本内部需要小数时由 named scripted value 转换。
- 离散枚举用单一整数或 mutually exclusive flag；不得两者并存表达同一状态。
- 资源若映射原生 prestige/piety/gold/state treasury，只通过 adapter。
- 日/月/年计数不得依赖浮点相等判断。
- “未知”不能用 `0` 代替；使用 `has_variable` 或明确 sentinel，并在 UI 隐藏精确数值。

### 8.3 事务边界

复杂动作采用：

1. 公共 trigger 验证所有前置条件；
2. 保存必要 scope；
3. 设置短期操作锁；
4. 调用若干原子 effects；
5. 写入记忆、dirty 和刷新请求；
6. 清除锁；
7. 失败恢复事件只处理确实可能中断的跨事件流程。

CK3 脚本没有通用数据库事务。设计必须避免“扣了钱但后续 scope 丢失”的跨事件链：关键扣费和状态变更应在同一个 effect 完成。

## 9. 统一调度

`core` 维护每个适用国家唯一调度器：

- 事件驱动：战争、继承、改革通过、危机阶段变化等；
- 月度：只处理截止期、锁和极少数活跃对象；
- 季度：AI 议程、危机推进资格、活跃改革；
- 年度：区域、阶级、联盟和阶段聚合；
- 五年：慢变量校准、角色代理轮换、旧记忆衰减。

调度规则以 `17A` 为硬约束。每个入口先检查适用政权、初始化、年份标记和幂等锁，再调用模块 effect。不得为每个角色注册独立年度全量计算。

## 10. DLC 与本体适配

### 10.1 必须包装的能力

所有下列调用先走 `minghm_adapter_*`：

- 是否拥有“溥天之下”及目标 feature；
- 是否为天朝/行政等适用政体；
- 国家国库读取与增减；
- 功绩、任命、家产、贡属、局势参与组；
- 王朝周期状态；
- 大工程、火器产能及其相关本体资源。

### 10.2 降级原则

- 缺少必须 DLC：不初始化依赖玩法，显示一次清楚提示，不刷错误事件。
- 可选原生界面不可用：使用事件/decision 降级，不复制整个本体 GUI。
- 本体键变化：只改 adapter 和兼容检查，不改上层机制变量。
- 版本未确认：启动调试报告显示 `unsupported_version`，但不自动破坏存档。

## 11. 事件与 scope 安全

- 每个事件头部注释根 scope、必要 saved scopes、触发入口和幂等要求。
- immediate 中先验证根 scope；可能失效的人物使用存在性检查和替代选择。
- 长链不能假定原人物仍活着、仍在同一宫廷或仍拥有头衔。
- 主国家状态挂在稳定主头衔/局势载体，不挂玩家当前人物。
- 区域状态挂稳定 region anchor title，不挂临时总督或县令。
- 危机 participants 用 situation group/聚合槽，关键人物不超过性能上限。
- `random_*` 必须先有可解释候选池；空池有后备结果。
- `trigger_event` 链必须登记，隐藏事件同样需要防重复锁。

完整模板见 `18B`。

## 12. AI 编码要求

### 12.1 AI 只能知道什么

AI 读取公开状态、自身资源、组织记忆和按可见度修正的估计。禁止读取：

- 玩家不可见且该 AI 无情报来源的精确危机隐藏值；
- 未来随机结果；
- 未被发现的其他组织承诺；
- 用全世界角色扫描得到的最优代理人；
- 为玩家专设的事件分支标签。

### 12.2 公共效用接口

选项 AI 权重应调用统一 value，至少纳入：

```text
material_gain
+ institutional_goal
+ survival_security
+ legitimacy_effect
+ ally_commitment
+ route_consistency
- resource_cost
- implementation_risk
- betrayal_cost
- uncertainty_penalty
```

各模块只提供分项，不自行发明另一套总效用。难度修正可以改变风险容忍和误判幅度，不能凭空给资源或读取隐藏真值。

## 13. 本地化与玩家信息

- 简体中文是首要编写语言；文件 UTF-8 with BOM 以符合 CK3 本地化加载习惯。
- 逻辑脚本 UTF-8；是否 BOM 按本地已验证惯例统一，不混用。
- 每个玩家选项显示：即时成本、主要支持者/反对者、执行风险、承诺和可能方向。
- 不显示 AI 的精确隐藏效用，不把未知信息冒充精确数。
- tooltip 使用公共 scripted localization 或值接口，禁止复制公式。
- 术语必须使用字典固定译名；C/P 编号只用于调试，不直接充当玩家叙事。
- 引用本体 `GFX_*` 资源前记录来源和 DLC；引用参考模组资产只记录可合法复用边界，不把原文件上传仓库。

## 14. 注释与代码风格

公共定义头部注释至少包含：

```text
# Purpose:
# Owner module:
# Input scopes/parameters:
# Writes:
# Reads:
# Called by:
# Frequency:
# Failure behavior:
# Evidence: S1/S2/D/R + document reference
```

- 统一四空格缩进；同类块顺序固定。
- 注释解释原因、边界和不变量，不逐行翻译语法。
- 不保留注释掉的大段废代码；版本历史交给 Git。
- 阈值通过 `minghm_const_*_value` 或集中 trigger 命名，不散落 magic number。
- 日志加 `[MINGHM][MODULE]` 前缀；正常游戏不输出高频日志。
- 暂未完成逻辑使用明确的 `BLOCKED` 注释并让测试失败，不以空 effect 假装完成。

## 15. AI 接收一个编码任务时的最小上下文

每次向代码生成 AI 提供：

1. 本次切片的可玩验收场景；
2. `18` 总规范；
3. `18A` 中实际使用的数据项和接口，而非随意改写全表；
4. `18B` 中对应模板；
5. 相关机制文档；
6. `13/13A` 中需要的本体接口与兼容边界；
7. `17A` 的频率和对象上限；
8. 允许改动的文件清单；
9. 当前 schema version 与迁移要求；
10. 必须运行的静态检查和实机测试。

如果上下文不足，AI 应停止并列出缺少的接口，不得用猜测补出本体 effect 名。

## 16. 每次代码提交的范围

一个提交应是可独立验证的垂直切片，例如“创建土地清丈改革并走完提议—试点—年度结算—反制—结果”。它可以同时包含事件、effect、trigger、本地化和测试，但不得混入另一个无关机制。

提交说明必须列出：

- 新增/修改的公共接口；
- 新增持久变量和 schema 是否变化；
- 性能频率与最大对象数；
- 与本体/参考模组的接口依据；
- 手动测试路径和观察结果；
- 已知但不阻断发布的限制。

## 17. 静态验收

至少检查：

- 所有新键具有 `minghm_` 或 `GFX_minghm_` 前缀；
- 事件 ID 唯一且属于模块号段；
- 无本体同路径覆盖，允许清单外覆盖为零；
- 本地化 key 存在，无明显孤儿或缺失；
- 公共接口在字典登记；
- 非所有者没有直接写入持久变量；
- 高频 on_action 不包含 world/character 全扫描；
- 所有活动对象有上限、冷却、锁和清理；
- 概率候选池为空时有降级；
- 旧存档缺变量时可初始化或迁移；
- error.log 无新增解析错误和未定义 effect/trigger。

## 18. 实机验收

每个切片至少测试：

1. 新游戏初始化一次且只一次；
2. 保存、读取后状态一致；
3. 主人物死亡/继承后国家状态仍在；
4. AI 能独立进入、推进、失败和退出机制；
5. 玩家可从 UI/事件理解成本与反制；
6. 极端值被 clamp，scope 丢失不破坏流程；
7. 同一 on_action 重复触发不重复结算；
8. 缺 DLC/不适用政权不产生错误刷屏；
9. 与一项相关原生机制联动正确；
10. 性能进入 `17A` 的对应测试层。

## 19. 禁止清单

- 禁止直接复制参考模组代码后批量改前缀；只借鉴经文档确认的结构和接口。
- 禁止覆盖本体通用 `on_actions`、`gui`、`events` 文件来插入功能。
- 禁止以每月全角色遍历模拟人口或阶级。
- 禁止每个省保存八类集团的全部七维矩阵；区域保存八轴，全国保存八集团。
- 禁止让阶段 P0—P9 成为线性科技树或历史必然路线。
- 禁止把资本主义、革命、官僚化或工业化写成无反作用的永久增益。
- 禁止给 AI 免费资源抵消其决策缺陷。
- 禁止事件 option 直接改十余变量而不经过命名 effect。
- 禁止在 GUI 每帧重算全国聚合。
- 禁止未登记的全局变量、事件号段、资源键和兼容覆盖。

## 20. 完成定义

一个机制只有同时满足以下条件才算“已实现”：

- 玩家入口、AI 入口、失败路径和退出/清理路径齐全；
- 源状态、缓存和界面显示没有多重真值；
- 存档、继承、scope 失效与缺 DLC 均有行为定义；
- 支持者、反对者、物质成本和长期反制均进入结算；
- 与至少一个原生 CK3 机制发生可见交互；
- 通过静态、功能、观察局和回归测试；
- 数据字典、事件登记和实现反查索引同步更新。
