# 施工版重构原则与缺口审计

> 本文回应“现有策划脱离本体、DLC 和参考模组”的问题。此前 `04`—`21A` 保留为设计依据与源码索引；从 `22` 起，所有文档按《变身大明》伴生扩展 Mod 的实际施工口径重写。

## 1. 产品形态冻结

本项目不是独立大明 Mod，也不是面向所有国家的通用社会模拟框架，而是：

```text
Crusader Kings III 1.19.0.6
+ All Under Heaven（溥天之下，强制 DLC）
+ 《变身大明》（Workshop 2713348525，强制前置 Mod）
+ 本项目（伴生扩展，最后加载）
```

本项目只扩展《变身大明》的 `h_greatming` 国家体系。不开《变身大明》不支持；没有“溥天之下”不支持；没有大明皇帝时逻辑休眠。

## 2. 不再采用的旧假设

| 旧假设 | 问题 | 施工版决策 |
|---|---|---|
| 可以独立创建“大明社会转型”框架 | 缺少实际大明头衔、政体、职位、地图和美术 | 直接依赖《变身大明》，读取 `h_greatming` 及其明制对象 |
| 可兼容所有天朝/官僚国家 | 扩大触发面、平衡和性能成本，违背用户范围 | 只对 `h_greatming` 及其内部明制角色生效 |
| 缺 DLC 时降级运行 | 需要维护第二套国库、家产、功绩和工程 | DLC 缺失时不启动；不做降级玩法 |
| 自建完整官职、科举、政体 | 与《变身大明》重复并冲突 | 读取参考 Mod 的流官、职位、考成和政体；仅增加结构反馈 |
| 自建大明 3D/建筑模型 | 成本高、风格和动画风险大 | 3D 全部直接引用本体/DLC/《变身大明》已加载资源 |
| 为兼容不使用参考 Mod 数据 | 会失去最可行的制度载体 | 明确成为伴生扩展，但不覆盖或重定义参考 Mod 数据库键 |
| 先写宏观变量，再寻找游戏载体 | 容易产生无法展示或无法驱动玩法的“影子系统” | 每项状态先指定原生/参考载体，再决定是否补变量 |
| situation 一定适合所有系统 | situation 需要类型、参与组、阶段、UI，成本不低 | 只用于全国长期转型；改革/案件/危机优先固定槽和事件/故事周期 |

## 3. 已有资料的价值与不足

### 3.1 仍直接保留

- `01`、`02`、`02A`：参考 Mod 文件、ID 和事件调用索引；
- `07`—`12B`：参考 Mod 已实现机制的源码还原；
- `13`、`13A`：本体/DLC 能力与覆盖风险；
- `14`—`17A`：历史唯物主义模型、长期博弈、AI 和性能原则；
- `18`—`18B`：命名、状态所有权和脚本模板；
- `20`：史料与文学的有限用途。

### 3.2 必须重做

| 缺口 | 现状 | 新文档必须补齐的证据 |
|---|---|---|
| 实际依赖 | 只写 DLC，没有冻结参考 Mod 前置 | descriptor、加载顺序、缺失行为 |
| 明制范围 | 使用泛化“适用国家” | `h_greatming`、`is_GM_emperor`、内部角色精确触发器 |
| 机制载体 | 多为“建议使用 situation/变量” | 每机制的本体/参考对象、路径、键名和不采用项 |
| 数据定义 | 已有通用字典，但未绑定参考 Mod | 哪些读原变量/特质/职位，哪些新建 `minghm_*` |
| 调用入口 | 有频率原则，缺准确挂接 | 使用的 on_action、根 scope、过滤和写入顺序 |
| UI | 有信息架构，缺具体复用途径 | 原生 situation/decision/event/参考 hub 的选择和禁止覆盖 |
| 美术 | 有原则，缺资源级清单 | 3D 直接引用对象、2D 复用池、新画上限 |
| 兼容 | 只强调零覆盖 | 与《变身大明》同启时的键/路径/变量/事件/加载顺序审计 |
| 施工拆分 | M0—M8 仍较抽象 | 每批新增文件、定义、变量、事件、测试和验收 |

## 4. 施工证据等级

从 `22` 起使用：

| 标记 | 含义 | 可否直接交给代码 AI |
|---|---|---|
| `V-B` | 本体 1.19.0.6 本地脚本已定位 | 可以，仍需实机加载验证 |
| `V-D` | “溥天之下”脚本/资源已定位 | 可以，前提 DLC 强制启用 |
| `V-M` | 《变身大明》脚本/对象已定位 | 可以，前提参考 Mod 强制启用 |
| `N` | 本项目新增定义，载体语法有已定位样板 | 可以按样板实现 |
| `P` | 需要最小技术探针才能决定 | 只能先写探针，不得扩展完整机制 |
| `X` | 明确不实现或不复制 | 代码 AI 不得自行补做 |

“可行”至少需要一个 `V-B/V-D/V-M` 载体和明确的新增范围；只有概念公式而无载体的条目不得进入施工队列。

## 5. 三类复用

### 5.1 直接读取，不复制

用于《变身大明》已有：

- `h_greatming` 和 `is_GM_emperor`；
- `sansizhi_government`、`weisuozhi_government`、`zongfan_government`、`tusizhi_government`；
- 流官特质、职衔、品级、军功、朋党、皇权、锦衣卫、宗藩、属国等已有状态；
- 已定义建筑、地产、兵种、GUI、纹章、服饰和模型资源。

本项目通过 trigger/value 读取，绝不复制同键定义。

### 5.2 新增带前缀的伴生层

用于参考 Mod 没有的：

- 全国级阶级力量聚合（区域生产已由 `24` 明确弃用）；
- 改革执行、承诺、反制和长期记忆；
- 资本形成、工业闭环、劳动关系和阶段变化；
- 明制专用的危机导演与 AI 效用；
- 必要的独立 decision、事件、轻量 situation 和 modifiers。

所有键使用 `minghm_*`；不修改参考 Mod 的原始文件。

### 5.3 只借语法，不接状态

某些参考机制可运行但不适合作为事实源，例如全人物扫描、过量人物记忆、大型 GUI 覆盖和逐件地契。只借其事件/interaction/effect 语法，不调用其高成本状态层。

## 6. 唯一事实源冻结

| 事实 | 唯一来源 | 本项目是否缓存 |
|---|---|---|
| 大明是否存在 | `title:h_greatming` 是否有 holder | 否 |
| 谁是大明皇帝 | `title:h_greatming.holder` / `is_GM_emperor` | 否 |
| 是否属于明制内部 | 皇帝或 `top_liege`/`employer.top_liege` 为大明皇帝 | 仅临时 scope |
| 官员职位、品级、朋党、军功 | 《变身大明》特质、职位、关系、变量 | 只做年度聚合，不复制人物事实 |
| 私人金钱 | CK3 character gold | 否 |
| 国家国库 | 《变身大明》/DLC 已启用 treasury | 只保存预算承诺，不复制余额 |
| 功绩、影响力、家产 | DLC 原生资源/对象 | 否 |
| 大明社会结构 | `title:h_greatming.var:minghm_*` | 是，项目唯一写入 |
| 区域结构 | 8 个登记 title anchors 的 `minghm_*` | 是 |
| 改革、项目、危机 | `h_greatming` 固定槽 | 是 |
| 外国状态 | CK3 原生关系、战争、贡属 | 不保存本项目宏观变量 |

## 7. “只在明制下生效”的硬规则

1. 所有玩家 decision 的 `is_shown` 先检查大明皇帝或被授权的明制官员；
2. 所有事件 trigger 必须能追溯到 `h_greatming`；
3. 所有周期入口第一层过滤必须是大明皇帝/明制内部角色；
4. 所有全国变量只写 `title:h_greatming`；
5. 区域锚点必须属于当前大明领域或登记的明制行政区域；
6. 外国统治者可以作为战争、贸易、朝贡或外交事件目标，但不初始化 C1—C8、阶段、改革或项目；
7. 大明灭亡/标题无 holder 时，调度停止，活动槽进入休眠；
8. 大明复国并重新获得 `h_greatming` 时，从该标题原状态恢复或按 schema 初始化；
9. 参考 Mod 的清/八旗政体不是本项目运行主体，只能成为外部压力或敌对目标；
10. 不做“适用于所有 celestial_government”的宽泛判断。

## 8. 兼容施工红线

- 不定义 `h_greatming`、`is_GM_emperor` 或任何参考 Mod 已有键；
- 不创建与参考 Mod 同相对路径的文件；文件名全部 `minghm_*`；
- 不覆盖 `window_character.gui`、`hud.gui`、`window_council.gui` 等通用窗口；
- 不重写参考 Mod 的 on_action 文件，只用新增文件向已存在 on_action 挂带前缀的子 on_action；
- 不直接写 `huang_quan`、`jungong`、朋党等参考 Mod 内部状态，除非专项文档确认公开写入 effect；默认只读；
- 不复制 3D、纹理或音频二进制到本项目；通过依赖直接引用资源键；
- 不修复参考 Mod 的通用缺陷，除非建立独立、可关闭的兼容补丁且用户明确要求；
- 不以本项目加载顺序覆盖参考 Mod 同键来“兼容”。

## 9. 新施工文档结构

| 文档 | 内容 |
|---|---|
| `22A` | 依赖、明制触发器、状态载体、初始化/休眠/恢复、on_action 挂接 |
| `23` 系列 | 皇权、内阁、官僚、朋党、监察、案件的逐机制实现 |
| `24–26` 系列 | 轻量数值边界、全国阶级/阶段、危机、抽象工业化与革命；财政、土地、市场和区域生产不实现 |
| `25` 系列 | 军事、边疆、危机、改革、革命和阶段的逐机制实现 |
| `26` 系列 | UI、美术/3D 复用、文件清单、加载兼容、性能和测试 |
| `27` | AI 可执行总施工顺序与逐批验收清单 |

每个机制条目必须写：玩家循环、原生/参考载体、读数据、新数据、入口、effect、AI、UI、美术、性能、失败和测试。

## 10. 本阶段完成标准

- 产品被定义为《变身大明》伴生扩展，而非独立/通用框架；
- 强制依赖、加载顺序和无前置时行为明确；
- 明制内外边界有可编码 trigger；
- 参考 Mod 已有事实与新项目变量不再重复；
- 后续每个机制都必须落到本体/DLC/参考 Mod 的具体载体；
- 3D 和绝大多数 2D 采取直接引用，不再把“未来画美术”当实现前提。
