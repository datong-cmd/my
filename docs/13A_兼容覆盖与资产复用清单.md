# 兼容、覆盖与资产复用清单

> 文档性质：参考模组冲突审计 + 新 Mod 文件边界 + 美术复用策略  
> 快照：参考模组 `2713348525` 与 CK3 `1.19.0.6` 本体同相对路径比较  
> 关联文档：技术底座见 [13](13_CK3本体与溥天之下技术映射.md)，参考模组全文件见 [01](01_参考模组源码文件索引.md)，资产分类见 [03](03_参考模组资产索引.md)。

## 1. 审计结论

参考模组与本体存在 **47 个完全相同的相对文件路径**：`common` 18 个、`gfx` 16 个、`gui` 10 个，另有事件、本地化和字体各 1 个。真正危险的不是文件数量，而是其中包含战争交互、授予头衔、议会职位、派系、角色窗口、HUD、事件窗口、军队指挥官窗口、字体和通用图标等全局入口。

**S1：** 参考模组 `descriptor.mod` 没有声明 `replace_path`，因此这些同路径文件不是启动器层面的整目录替换；但数据库键、GUI 类型、资产名或同名定义仍可能覆盖本体或与其他模组发生加载顺序冲突。

**D：新 Mod 默认采用“零通用文件覆盖”原则。** 所有新增数据库键、事件、变量、GUI 类型、图标名和本地化键使用统一项目前缀。只有引擎确实没有追加接口，且该功能对 MVP 不可缺少时，才允许在独立兼容模块中覆盖一个本体定义。

### 2.3 图形、字体和本地化

同路径图形包括通用黄金、士兵、威望等级图标，征召兵/骑士插画，以及 `barbershop.txt`、`ingame.txt`、`main_menu.txt`、`all_buildings.asset`、`naming.txt` 等全局资源注册文件。另有 `fonts/fonts.font` 与一个身体部位 custom loc 文件。

这些文件多数不是“新增一个图”这么简单，而会影响全局资源名、建筑模型登记、字体或主界面。新 Mod 不得原样复制，也不得用相同资源名替换本体图标。

## 3. 覆盖风险等级

| 等级 | 定义 | 示例 | 政策 |
|---|---|---|---|
| C0 | 全新增、前缀唯一 | 新事件、新修正、新工程、新局势 | 默认允许 |
| C1 | 读取本体接口或资源路径 | 使用国库、功绩、原生图标 | 允许，记录 DLC/版本依赖 |
| C2 | 向可合并数据库追加新键 | 新建筑、兵种、交互、决议 | 允许，做键名重复检查 |
| C3 | 覆盖单一本体数据库键 | 修改原生任命评分或政府条目 | 仅兼容模块，逐版本 diff |
| C4 | 覆盖通用 GUI、战争、派系、议会或全局资源文件 | `hud.gui`、`00_war.txt` | MVP 禁止 |
| C5 | `replace_path` 整目录替换 | 整个 `common/governments` | 禁止 |

每个 C3 文件必须在仓库维护一条记录：本体原始版本、覆盖原因、修改的数据库键、上游 diff、已测试模组组合和移除方案。没有记录即视为缺陷。

## 4. 参考模组可借鉴与不可直接搬运的边界

### 4.1 可以借鉴

- 机制拆分方式、事件节奏、交互流程和状态机思想；
- 已验证的 CK3 trigger/effect 组合，前提是按 1.19.0.6 本体重新核对；
- 明代官制、称谓、地方单位和玩法主题；
- 参考模组如何用人物、头衔、变量和 GUI 表达非封建政治关系；
- 性能问题和兼容问题本身，作为反面验收样例。

### 4.2 不直接复制

- 未取得许可的原创图像、字体、音频、模型、文本和大段脚本；
- 参考模组覆盖的本体通用文件；
- 与本体同名的资源、事件命名空间、变量和数据库键；
- 已被 DLC 原生系统取代的平行国库、官职、朝贡或候选池；
- `.txty`、`.txtz` 等非标准后缀备份文件；它们通常不会被游戏数据库加载；
- 依赖旧版本隐式行为但没有明确调用入口的脚本。

**R：法律与发布边界。** 技术上能读取和引用不等于取得再分发许可。新 Mod 发布包只包含团队原创文件；本体/DLC 资产通过游戏内路径引用，不复制进仓库；参考模组原创素材只有在作者明确授权后才能随包分发。

## 5. 本体与 DLC 资产复用策略

### 5.1 复用原则

1. **路径引用优先。** GUI、事件和数据库直接引用游戏已安装的图标、插画、模型和音效路径。
2. **不复制专有文件。** 仓库不存本体/DLC 的 `.dds`、`.mesh`、`.asset`、字体或音频副本。
3. **语义相符。** 官署图用于官僚制度，工坊/纸坊/陶窑图用于生产节点，不能为了省美术把同一图标泛用于所有资源。
4. **DLC 门槛一致。** 使用 DLC 专有系统和资产的内容只在 `has_tgp_dlc_trigger` 成立时初始化。
5. **唯一自定义名。** 自制图标、GUI 类型和 sprite 名全部加项目前缀。

### 5.2 现成资产池与建议用途

| 资产池 | 本体/DLC 来源 | 推荐用途 |
|---|---|---|
| 东亚家产建筑 | `common/domiciles/buildings/00_chinese_estate_buildings.txt` 对应图标/插画 | 书院、档案、税务、吏员、工坊、仓储和家族制度资本 |
| 天朝政体与部院 UI | All Under Heaven 原生窗口和图标 | 国库、功绩、影响力、任命与部院，不另做主窗口 |
| 科举活动资产 | `activity_imperial_examination` | 科举、策试、专业考试事件 |
| 巡察活动资产 | `activity_survey` | 巡按、考成、勘灾、边防巡阅 |
| 大型工程资产 | 长城、运河、强化首都、军营工程 | 漕运、海塘、军器局和基础设施工程模板 |
| 火药与东亚兵种 | 火药库、火枪手、火枪骑兵、火箭车、火炮 | 军工能力和早期军事现代化 |
| 天朝朝贡资产 | `tributary_celestial` 和册封特权 | 朝贡、册封、贸易和保护义务 |
| 王朝周期资产 | situation 阶段、集团、催化剂和历史记录 | 新社会转型局势的技术范本，不复用同名键 |

### 5.3 美术缺口

本体资产足以完成 MVP。真正可能需要原创的仅有：

- 社会转型局势的主图和四至六个政治联盟图标；
- 阶级力量、组织度、劳动关系和工业能力的少量概念图标；
- 后期标准化工场、蒸汽动力或革命组织的事件插画；
- 独立总览窗口若最终确有必要时的装饰框。

在机制确定前不制作大批兵模、建筑模型或肖像配件。后期工业化优先通过事件插画、图标、建筑和军队修正表达，避免 3D 资产成为开发瓶颈。

## 6. 安全扩展方式

### 6.1 使用唯一前缀

下列只是编码示例，最终前缀在 `18` 文档统一冻结：

```text
事件命名空间：minghm
数据库键：     minghm_*
变量：         minghm_*
全局变量：     global_var:minghm_*
修正：         minghm_*_modifier
本地化：       minghm_*
GUI 类型：     minghm_*
资源名：       GFX_minghm_*
```

不要使用 `yan_*`、`GM_*`、`ming_*` 这类可能与参考模组或其他明代模组冲突的宽泛键。

### 6.2 用桥接层隔离本体变化

所有本体/DLC 专用判断集中在少量 scripted triggers/effects：

```text
minghm_has_required_dlc_trigger
minghm_is_celestial_state_trigger
minghm_get_state_treasury_value
minghm_add_state_treasury_effect
minghm_is_in_dynastic_crisis_trigger
minghm_recalculate_military_industry_effect
```

业务事件只调用桥接层。游戏补丁改变底层键时，只修改桥接文件，不全仓搜索替换。

### 6.3 不覆盖也能实现的典型需求

| 需求 | 安全做法 |
|---|---|
| 限制某类战争 | 自定义 CB 的 `can_use`、目标关系和事后事件，不改 `00_war.txt` |
| 任命考成加分 | 履历变量/角色修正 + 自定义 scripted value；必要时单键 C3 覆盖 |
| 新六部职位 | 追加带前缀职位或复用原生部院职位，不复制整个 council 文件 |
| 阶级派系 | situation 参与集团和自定义 faction 键，不改 `00_factions.txt` |
| 显示国家状态 | situation UI、决议提示、重要行动、scripted GUI 独立窗口 |
| 人口性能优化 | 年度调度与候选范围过滤，不改 guest/courtier 管理文件 |
| 新工程 | 追加 `great_project` 类型，不打包 `_great_project_types.info` |
| 新兵种 | 追加 MAA 键与独立图标引用，不覆盖整个兵种文件 |

## 7. 与参考模组同时启用的政策

参考模组本身覆盖大量通用 GUI 和脚本，并实现与新项目重叠的国库、官制、朝贡、财政和军事系统。即使新 Mod 避免覆盖，两个模组同时初始化同一大明政权也会产生语义冲突。

**D：正式支持矩阵如下。**

| 组合 | 支持级别 | 说明 |
|---|---|---|
| CK3 1.19.0.6 + All Under Heaven + 新 Mod | 正式支持 | 主测试环境 |
| CK3 1.19.0.6 + 新 Mod，无 DLC | 不支持 | 给出依赖提示，不初始化 |
| 参考模组 + 新 Mod | 不支持 | 机制与全局覆盖重叠；只把参考模组当开发资料 |
| UI 大修 + 新 Mod | 尽力兼容 | MVP 无通用 GUI 覆盖，仍需组合测试 |
| 其他东亚政体大修 + 新 Mod | 未承诺 | 取决于是否覆盖天朝政府、契约和王朝周期 |

启动时可检查参考模组特有的全局标记、政府、决议或变量；检测到后只显示一次不兼容警告，不尝试自动删除对方状态。

## 8. 自动兼容检查

仓库后续应提供只读检查脚本，在每次发布前完成：

1. 列出新 Mod 与本体的同相对路径；
2. 列出新 Mod 与参考模组的同相对路径；
3. 扫描无项目前缀的事件 ID、数据库键、变量、GUI 类型和资源名；
4. 检测打包中是否包含 `.info`、`.txty`、`.txtz`、临时文件和本体/DLC 复制资产；
5. 对所有 C3 覆盖生成本体 diff；
6. 检查本地化键缺失、重复和编码；
7. 检查 DLC 专有入口是否有统一门槛；
8. 用 `-debug_mode` 查看 `error.log`，把新增错误视为发布阻断。

建议的构建门槛：C4/C5 覆盖数必须为 0；未登记 C3 必须为 0；复制的本体/DLC 二进制资产必须为 0；无前缀自定义键必须为 0。

## 9. AI 编码交接约束

后续把策划交给 AI 生成代码时，应把下列约束放在每个实现提示的开头：

```text
目标版本固定为 CK3 1.19.0.6，依赖 All Under Heaven。
只新增带统一前缀的文件和数据库键。
不得复制或覆盖通用本体 GUI、00_war、00_factions、00_council_positions。
不得声明 replace_path。
优先复用 celestial government、treasury、merit、influence、appointment、estate、
tributary contract、situation、activity、great project 和 gunpowder capacity。
若必须覆盖本体单键，先停止并输出：覆盖原因、目标键、最小 diff、兼容风险和替代方案。
不得把参考模组原创资产或大段脚本复制进项目。
```

## 10. 验收清单

- [ ] 新 Mod 与本体同路径清单为空，或仅含已登记 C3；
- [ ] 没有 `replace_path`；
- [ ] 没有通用 HUD、角色、议会、事件、军队或物品栏窗口覆盖；
- [ ] 没有复制本体/DLC/参考模组的专有二进制资产；
- [ ] 所有自定义键、变量、事件、GUI 和资源名使用冻结前缀；
- [ ] DLC 资产与系统入口都有依赖门槛；
- [ ] 与参考模组同时启用时给出一次不兼容说明；
- [ ] 所有 C3 覆盖有上游版本、最小 diff 和测试记录；
- [ ] 发布包不含 `.info`、备份后缀、分析脚本和临时文件；
- [ ] `error.log` 无新增解析错误、重复键和缺失资源。
